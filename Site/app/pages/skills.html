C / C++

6 Years experience with C++, 3 of which were in a professional setting.

I love working with C++, particularly C++11, as the language offers both a great deal of challenges as well as control. When I work on a project with C++ I feel like I am taking all of my tools out of the shed, whereas with other languages I feel I'm just approaching the project with a toolbox. Generally this is my preferred language for personal projects. I frequently look up blogs on interesting topics and libraries using C++ as this is a language I love to explore. Additionally I have been following the C++ standardization process quite closely over the last couple of years and can't wait to start see the C++14 and C++17 features as they roll out over the next few years.

Additionally I have experience working with pure C on Linux machines (specifically Ubuntu), Windows running with Cygwin and on an embedded system called the ColdFire. Furthermore my Windows 8 projects have dealt with Microsoft's C++/CX extensions.

Note: All of the following code is part of a work in progress Windows8 music player and is being actively worked on. Expect to see the odd TODO.
  • MPEG4Parser.cpp:

    The core class of a parser I wrote for the MPEG4 container. The parser was written to the ISO specification and handles all of the standard box types as well as several of the iTunes specific boxes. Root of project is here: MPEG4 Parser

  • MusicProvider.cpp:

    A class that is responsible for gathering songs and providing them to any external system that asks for them. Will load songs from both a local cache and from disk while also handling the cases of duplicate songs (ie. same song 2 separate files) and songs that have multiple copies in different formats (ie. a song that has an mp3, AAC and ALAC copy).

  • ALACMFTDecoder.cpp:

    A media transform for use with Windows Media Foundation that allows for WMF to play ALAC music that is contained within an MPEG4 container. Currently functional and successfully playing a wide sample of ALAC encoded music from my library. Uses Apples ALAC decoder.

  • InterfaceWrapper.hpp:

    A simple general purpose wrapper for objects of a similar interface. Intended for use in coercing two classes with the same interface but no common base-type into a single type that can easily be passed around.

Haxe, Javascript

Haxe and Javascript are other languages I've used but do not consider myself experienced enough with to say that I am proficient.

Haxe: I was exploring this language as an alternative to AS3 by using OpenFL. Haxe's ability to compile into several other languages such as AS3, C++, C# and JavaScript seems to make it quite flexible and I expect it would be fairly simple to find a means to integrate Haxe into an existing system. What interested me most was that it had better language features than AS3, in particular it was statically typed, but could easily be converted into AS3 allowing it to be compiled into a SWF. That SWF could then easily be consumed by either Adobe Air or Scaleform which means that switching to using Haxe instead of AS3 would not require much retooling work of an existing production pipeline. In the future I really want to sit down and work on a small game or other application using Haxe and either OpenFL or Flambe.

Javascript: General experience through Unity (unity-script) as well as hacking on simple web based projects and Node.js based tools. My knowledge is not extensive but I can work with the language without much difficulty. Someday I would like to build a web app that uses JavaScript and Node.js to expand my knowledge of the tools. In particular I want to learn more about asm.js and how native code performance can be integrated into the web browser to provide a richer client experience. By richer I mean something comparable to a native application in terms of performance with fancy webGL rendering effects.
